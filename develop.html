INTRODUCTION

Welcome to the PWNHS Development Kit. This page covers basic requirements and features of the PWNHS website project.

What the PWNHS Development Kit is

To open up more avenues for engagement with the PWNHS community as a whole, we have opened up the PWNHS site project for tweaking by those in our organisation who are particularly code-savvy. The Development Kit provides basic structure for how the site should behave and establishes a path for anyone to help improve our website.

What the PWNHS Development Kit is not

This document does not hold proprietary APIs, Content Management Systems, of FTP clients. Developers are given access to static infrastructure, not server-side technologies.

GETTING STARTED

Project Base

PWNHS is a GitHub Pages project hosted (consequently) on GitHub. You can visit the project here. To explore the project and offer contributions, you must first fork and clone the PWNHS repository.

Forking

To fork the project, click the button below this block. Just save the fork to your account and you're done!

Fork

Cloning to Local

In order to edit your fork of the PWNHS repo, you have to clone the fork to your machine so that all of the code is locally avaliable. Cloning can be done in two ways - via GitHub's desktop application or via command line.

Cloning via Desktop App

Cloning via GitHub's desktop app is a one-click process. After downloading the app, simply go to your forked repository of PWNHS (with your GitHub username instead of USER): 

https://github.com/USER/pwnhs.github.io
and press Open in Desktop under the Clone or download button. This automates the entire process, and we suggest all but the most experienced take this route.

Cloning via Command Line

To create a local clone via command line, open up Terminal or Git Bash and type in git clone followed by the URL of your fork. You can find this URL by going to your fork of PWNHS, clicking the Clone or download button, and copying the URL found below the Clone with HTTPS block. Your command line will look like this: 

git clone https://github.com/USER/pwnhs.github.io.git
Press Enter. Check to make sure all objects are unpacked and the command line reads done.

To allow for upstreaming (pulling of changes from the Master repo to your fork), navigate to the location of your fork in the command line and type in git remote add upstream followed by the URL of the original PWNHS repo. Your command line will look like this: 
git remote add upstream https://github.com/pwnhs/pwnhs.github.io.git

Note

IMPORTANT NOTE: In order to submit pull requests to the Master, you must both fork and then clone the Master PWNHS repo. Only cloning it will not create a server-side branch of the repo, thereby not tying your clone to the Master and disallowing you the ability to merge with the Master branch. More information on forking a repo here and here.

DEVELOPER ALLOWANCES

Although we are more than eager to have you tweak and improve upon our site, some things better left alone or are only to be handled by the NHS core.

Do's


DO edit format, style, animations, frameworks

DO tweak expanded stylesheets, javascript, html

DO add style- and feature-based extensions

DO observe consistency across pages and styles

DO keep with established fonts and the color palette

Don'ts


DON'T edit content (text, pictures) of information

DON'T touch minimised files (they're dependencies!)

DON'T add infrastructure-altering extensions

DON'T generate hyper-independent structures

DON'T neglect dependencies or add redundancies

ISSUES & SUBMISSIONS

A community is built on discussion and contribution. We want those of PWNHS who choose to aid in our project to develop a mutually dependent, beneficial collective. That's why opening issues and submitting pull requests are so important.

Issues

Issues are a way to point attention to and/or discuss a bug, improvement, or anything else of substance. If you see any place for improvement or have something to bring up, open up an issue on the PWNHS repo. More on issues here.

Submissions (Pull Requests)

A pull request is a way to submit features or fixes of the project you have made on your fork to the master branch. Pull requests are the most important part of your forking process, as they show what contributions you have to propose.

To make a pull request, go to your fork of PWNHS on GitHub: 

https://github.com/USER/pwnhs.github.io
where you will see a banner noting that you have recently pushed a new branch of the PWNHS project Master. Click on Compare & pull request. After making a title and filling the description with information regarding what you have added/changed/removed and why, press Create pull request.

Note

IMPORTANT NOTE: Though the vast majority of your legitimate contributions will get in, there will be the occasional pull request that does not meet our standards. To make sure that your contribitions have the optimal chance of acceptance, make sure they correspond with our developer allowances and there is nothing in your pull request that does not directly correlate with the betterment of the PWNHS site.

PROJECT INFRASTRUCTURE

The PWNHS site is built on a proprietary Molekule (Hafiz, 2013) architecture based on the open-source Blocks (Sheeran, 2011) method (used in molekuleapp). Molekule finds an advantage in web development through its enhanced modularisation of typical site components and SpecGen custom subclass styling. Molekule uses two fundamental bases, "mass" and "unit", to generate modules that come to make up the entire site. Just as cells make tissues that make up organs, classic styling makes up masses that make up units.

"mass" Structures

Masses can be thought of as the various elements, or building blocks, of a site. These include things like paragraph sections, images with captions, rows of icons, etc. Instead of having to code an element manually, a mass provides a preset that describes the layout, structure, and function of the element for you. Confused? Here's an example of code that describes the panel of Upcoming Projects and Volunteer Opportunities on the PWNHS main page: 

<div class="container">
    <div class="row animSpeedFast voffset">
        <a class="keep" href="http://act.alz.org/site/TR?fr_id=9319&pg=entry">
            <div class="col-sm-4 expansion">
				<div class="panel">
                    <div class="panel-heading">
				        <h3 class="mg-clear  text-right" style="font-weight:600">
                        Walk to end Alzheimers
                        </h3>
                    </div>
                    <div class="panel-body">  
                        <img src="img/Final_walk_TALK.png" class="img-responsive" />
				    </div>
                </div>
            </div>
        </a>
    </div>
</div>
Unless you really love <div>s, manually creating this is not ideal. A mass, then, makes the entire preset for you, complete with the five inset div tags, numerous classes, and header styles, so that all the user has to do is add content.

Most of the classes imbeded in masses, such as col-sm-4 in our example, are defined by Bootstrap styling. More custom classes exclusive to Molekule are described in the general stylesheet. In our example, these custom classes include voffset, panel, and mg-clear.

Because we expect our developers to modify the style of the site (as accorded by our developer allowances), and not the content, we do not have a list of all Molekule masses on this Docs page. You are supposed to edit or add/modify to elements already-existing mass classes, which can be found in ./style.css.

If you would like to obtain the full documentation and dependencies of Molekule (which includes all "mass" structures) for your own projects, just say hi!

"unit" Structures

Much broader in scope and easier to apply, a unit can be thought of as the larger frame in which masses are held. PWNHS's landing page, for instance, is full screen, light, and parallaxed - a format allowed by the following structure: 

<div class="unit unit-fill-screen b-parallax bg-IMGx bgc-white l-unit" id="florida">

    <!-- landing page content -->

</div>
The styling for the relevant unit classes (unit, unit-fill-screen, l-unit) are as follow: 
.unit{
	width:100%;
	clear:both;
	background: 50% 50% no-repeat;
	padding:0 50px;
	-webkit-background-size: cover;
	-moz-background-size: cover;
	-o-background-size: cover;
	background-size: cover;
	position:relative;
}

.unit-fill-screen{
	padding-top:0;
	position: relative;
	height: inherit;
}

.unit-fill-screen > .container{
	position:relative;
	top:50%;
	transform:translateY(-50%);
	-webkit-transform:translateY(-50%);
}

.l-unit{
	color:rgba(0,0,0,.5);
}
Ultimately, then, units allow for the encompassed styling of the container in which masses are held.

What do I do with this?

We suggest you play around and edit the styling of various "mass" and "unit" structures to create something you find appealing and think would better the PWNHS website. You can really do anything, as long as you upkeep the infastructure we've built and observe our allowances.

EXTENSIONS

Several extensions, including Typed and Prism, are already installed into the PWNHS project as add-ons to improve the functionality and appearance of the site. If you wish to add an extension, reference the do's/don'ts of extensions below.

Do's


DO add stylistic extensions (css, js, svg, fonts)

DO add static extensions (don't require user input)

Don'ts


DON'T add systematic extensions (Jekyll exception)

DON'T add dynamic extensions (php, sql, MySQL)



Want to Use Jekyll?

Though we have not had much success with it, if you want to add a "toaster for websites" like Jekyll,

1) Make sure you have experience with Ruby, Python, and have used Jekyll(-like) before,

2) Have backed-up your branch of the project (it is suggested you create a totally different branch for the implementation of Jekyll), and

3) Modify the ._config to exclude biases, blog feeds, and correctly point to the PWNHS server.

Check out Jekyll's site for more information.

FONTS & ICONS

If you haven't noticed yet, we really like icons. They're pretty, space-efficient, and quite useful in the construction of site content. Same goes for our fonts - we use only two for content, and our code fonts are system-based, requiring no calls.

FONTS

We use two fonts for site content - Montserrat, a free Google font, and Proxima Nova, a premium font stylistically similar to Montserrat. Montserrat is used for points of emphasis, whilst the minimalistic style of Proxima Nova is exceptionally made for body content.

Code fonts are described in the following manner: 

code{
    font-family: Monaco, "DejaVu Sans Mono", "Courier New", monospace ;
}
with Monaco obviously being the most prefered reference.

Code blocks are described in the following general way: 
code./*some class*/{
    font-size: __px;
    color: #__;
    padding: __px __px __px;
    border: __px solid #__;
    -webkit-border-radius: 3px;
    -moz-border-radius: 3px;
    border-radius: 3px;
    background-clip: padding-box;
    background-color: #__;
}
We require that you do not change our two content fonts, Montserrat and Proxima Nova, but you are more than welcome to configure code styles to your liking.

ICONS

This site mainly uses Font Awesome , with guest appearances of Linecons to fit in missing pieces. The general structure for the insertion of Font Awesome icons is: 

<span class="fa fa-some-icon"></span>
with the following output:
While you are only allowed to add more icon packs, not modify or remove existing ones, you are welcome to edit icon styles located in ./style.css or ./documentation/docs-style.css. These styles are described in a manner similar to the following example: 
.icon-md-lg{
	font-size: 45px !important;
}
Because the icons default to Font Awesome's styling, the addition of !important is imperative to override normal function.

ANIMATIONS

Animations make the world go round. Literally, in the case of our pageload spinner:

Static Animations

Right now, we only have one static animation - the spinning wolf, located at ./img/pageload-spinner.gif. If you would like, you can make a better version of this animation, but don't remove it or make it anything other than the West logo!

Embedded Animations

Embedded Animations are those written directly into the HTML, CSS, and JS of the PWNHS project. Currently, this includes Typed, colorFades, fadeIns, and expansions.

Typed is a framework that allows for the typing animations you see at the bottom of PWNHS Docs pages, and in this header above. It is generally used by first describing a Typed class in JS and then calling to it in the HTML. 

$(function(){
    $(".some-class").typed({
        strings: ["String_1", "String_2", "String_3"],
        loop: true,
        typeSpeed: 100,
        showCursor: true,
    });
});

<span class="some-class"></span>

You can directly modify the js and styling of Typed by editing ./scripts/typed/js/typed-code.js and ./scripts/typed/main.css, or you can play with existing scripts found between the <head></head> tags in the HTML of the docs pages.

colorFades

Color animations drive attention to otherwise idle objects - particularly to icons. The way PWNHS describes these animations is pretty simple, as exampled by the following HTML/CSS duo. 

<span class="colorFade"></span>


@keyframes colorFade { 
	0%   { color: #8597A4; }
	50%  { color: #fa575d; }
	100% { color: #8597A4; }
}

@-moz-keyframes colorFade { 
	0%   { color: #8597A4; }
	50%  { color: #fa575d; }
	100% { color: #8597A4; }
}

@-webkit-keyframes colorFade { 
	0%   { color: #8597A4; }
	50%  { color: #fa575d; }
	100% { color: #8597A4; }
}

.colorFade {
	animation:         colorFade 2.5s infinite; 
	-moz-animation:    colorFade 2.5s infinite; 
	-webkit-animation: colorFade 2.5s infinite;
}

We use -moz and -webkit prefixes to allow for optimal performance across all modern browsers. While it is possible to automate this process via JavaScript, we have found our current process is less time consuming. If you wish, you may pursue a js route; the PWNHS core development will focus on such optimisation when all major features of our bucket are completed.

fadeInDown

fadeInDown is used for the sticky navbar on the PWNHS core site. The animation is described by ./css/animate.min.css, and called by: 

<div class="fadeInDown animated"></div>

Expansion

Like colorFade, expansion animations improve the aesthetic of the PWNHS site and allow for subtle emphasis in crucial areas of our content when moused over. Also like colorFade, it may be possible to automate this with JavaScript, but that option is of low priority to the NHS core at this time. The animation is called in the HTML and its :hover class is described in the stylesheet. 

<span class="expansion"></span>


.expansion { 
    -webkit-transition: all .3s ease-in-out; 
} 

.expansion:hover {
    -webkit-transform: scale(1.02);
}

WEB APP

The PWNHS website contains support for its iOS and Android web (addToHomeScreen) app. Designing this was very easy, having only to set the viewport as static in a meta tag: 

<meta name="viewport" content="width=device-width, initial-scale=1, user-scalable=no">
And then allowing for iOS web app capability as a meta tag between <head></head>: 
<meta name="apple-mobile-web-app-capable" content="yes">
Android does sets the Web App styles by itself when a website is added to the homescreen, so there is no need to define android web app capability.

To make sure there is no unwanted status bar overlap on iOS, the following meta tag is also added: 
<meta name="apple-mobile-web-app-status-bar-style" content="default">

NATIVE OS BUILDS

If you want to build a native iOS, Android, or even Desktop app for the PWNHS site, please check out our tips for each OS below.

iOS - Swift

To make an app for iOS based off the PWNHS website (HTML), import UIKit and WebKit in ViewController.swift: 

import UIKit
import WebKit
call in an WKWebView webView controller: 
class ViewController: UIViewController {

    var webView: WKWebView!
    
    @IBOutlet var mainView: UIWebView!
    
    override func loadView() {
        super.loadView() // call parent loadView
        self.webView = WKWebView() // instantiate WKWebView
        self.view = self.webView! // make it the main view
    }
    
}
and then identify the webpage webView is to load: 
    override func viewDidLoad() {
        super.viewDidLoad()

        let url = NSURL(string:"http://pwnhs.org")
        let req = NSURLRequest(url:url! as URL)
        self.webView!.load(req as URLRequest)
    }

The full code will look something like:

//  ViewController.swift
//  PWNHS.app

import UIKit
import WebKit

class ViewController: UIViewController {

    }

    }
    
}

iOS + Android - Objective-C

Cross-platform development with Objective-C is a bit tricky. Though most of the code can be the same, you will have to create a interface wrap for each platform individually to avoid interfering UIs. Of course, you can just make a Java-based webView app for Android, but here we'll just outline the basic premise for the construction of a webView in Objective-C, as it can be relayed to Android through the NDK and to iOS independently.

First, import UIKit and set up UIWebView in viewController.h: 

#import <UIKit/UIKit.h>

@interface ViewController : UIViewController
@property (weak, nonatomic) IBOutlet UIWebView *webView;

@end
call in viewController.h in viewController.m and identify http://pwnhs.org as the URL to load: 
//
//  ViewController.m
//  PWNHS
//
//  Created by Plano West NHS.
//  Copyright Â© 2016 PWNHS. All rights reserved.
//

#import "ViewController.h"


@interface ViewController ()

@end


@implementation ViewController

- (void)viewDidLoad {
    [super viewDidLoad];
    NSString *fullURL = @"http://pwnhs.org";
    NSURL *url = [NSURL URLWithString:fullURL];
    NSURLRequest *requestObj = [NSURLRequest requestWithURL:url];
    _webView.delegate = (id)self;
    [_webView loadRequest:requestObj];
}

@end
And you're done!

Desktop - Electron

Yes, "Desktop" is not an OS. But if you choose to make a PWNHS application for desktop, we require you use Electron and its APIs to to create a wrap of the PWNHS site that is able to run on Windows, Mac, and Linux. Electron is very easy to use and creating applications with it is remarkably simple, so we'll let you check it out on your own.
